Claude Code Under The Hood
01

你面对的不是聊天框，而是一个 Agent Runtime

先从用户已经知道的事情开始：Claude Code 能读代码、跑命令、改文件。真正值得学的是，这些行为是如何被编排成一个可靠的运行时系统。

它到底在做什么

把它想成一个会循环工作的协作系统。用户提出任务，系统收集上下文，模型决定下一步，工具去执行，结果再回到模型，直到任务完成或进入恢复路径。

🧭
入口层

main.tsx 把配置、技能、命令、权限和渲染器都装起来。

🧠
思考层

query.ts 不是单次请求，而是一整套回合式执行循环。

🛠️
执行层

tools.ts 和 Tool.ts 把工具抽成统一能力协议，让模型能安全调用现实世界的动作。

💡
PM 关键洞察

AI Agent 产品不是“模型能力展示页”，而是把模型塞进一个有runtime、工具、权限和恢复逻辑里的产品系统。

先看一段真实代码

这段代码来自启动入口。它很像“后台提前备餐”：用户还没真正开始操作，系统已经把未来会用到的信息并行预热了。

Code 
// eslint-disable-next-line custom-rules/no-top-level-side-effects

Plain English

程序一启动，就先去读取受管配置，而不是等后面需要时再查。

它还提前去拿 Keychain 里的鉴权信息。

这么做的目的不是功能炫技，而是减少后续等待时间，让启动更快。

为什么启动阶段值得 PM 关心

很多团队会把启动时间当成工程优化细节，但在 agent 产品里它直接影响用户是否愿意把更多任务交给系统。启动越慢，用户越容易切回“我自己手动做”。

如果启动慢

用户会把 agent 当成“重型仪式”，只在不得不用时才打开。这样功能再强，也难以进入高频工作流。

如果启动快

用户会更愿意把小任务也交给 agent，长期看会形成更强的依赖和更高的工具使用密度。

⏱️
PM 视角

冷启动时间不是单纯的性能指标，它决定产品能不能进入用户的“顺手路径”。对 agent 来说，这几乎等同于留存门槛。

02

认识这套系统的演员表

学会和 AI 工具合作，首先要知道每个模块各自负责什么。这样你才能判断新能力应该放在哪一层，而不是什么都塞进 prompt 里。

文件树比大段文字更好懂

src/ 整套运行时都在这里
main.tsx装配启动、配置、渲染、命令和技能

🧩
为什么这很重要

对 PM 来说，capability packaging 决定功能是否可理解、可治理、可扩展。命令、工具、技能不是同义词，而是不同的产品层。

为什么 commands、tools、skills、plugins 不能混着讲

这四层都像“能力”，但它们解决的是完全不同的问题。很多 AI 产品越做越乱，就是因为所有新需求都被塞进同一个层级。

Commands

用户显式触发的入口。适合包装“我要做一件事”，比如 review、doctor、memory。

Tools

模型内部可调用的原子动作。适合包装“系统怎么做”，比如读文件、跑命令、开子代理。

Skills

可复用的流程知识。适合沉淀某类任务的稳定做法，让 agent 不用每次从头想。

Plugins / Bridge

面向生态和平台扩展。适合接入外部系统、IDE、团队能力和更长的产品边界。

03

一轮请求到底怎么跑完

这里是 Claude Code 的心脏。理解这个流程后，你就会知道为什么 Agent PM 必须关心恢复、预算、权限和状态，而不是只看 prompt 文案。

query.ts 的关键代码在说什么

Code 
export async function* query(
  params: QueryParams,
): AsyncGenerator<
  | StreamEvent
  | RequestStartEvent
  | Message
  | TombstoneMessage
  | ToolUseSummaryMessage,
  Terminal
> {
  const consumedCommandUuids: string[] = []
  const terminal = yield* queryLoop(params, consumedCommandUuids)
  for (const uuid of consumedCommandUuids) {
    notifyCommandLifecycle(uuid, 'completed')
  }
  return terminal
}
Plain English

这不是一个普通函数，而是一个会边执行边产出事件的异步生成器。

真正的活在 queryLoop 里，它会持续推进这一轮任务。

任务结束后，系统还会把这轮里消费过的命令标记成 completed，保证状态一致。

真正拉开差距的是恢复逻辑，不是第一下答得多漂亮

看看 query.ts 里这一段。它不是“请求失败就报错”，而是在认真区分什么错误值得拦住、什么错误应该继续恢复。对 agent 产品来说，这就是可用性本身。

Code 
const MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3

function isWithheldMaxOutputTokens(
  msg: Message | StreamEvent | undefined,
): msg is AssistantMessage {
  return msg?.type === 'assistant' && msg.apiError === 'max_output_tokens'
}
Plain English

系统明确知道：模型可能因为输出太长而中断，这不是罕见事故，而是常见运行条件。

所以它提前定义恢复上限和识别逻辑，避免把中间错误过早暴露给上层产品。

这相当于在产品里设计了“继续把这件事做完”的能力，而不是只会失败提示。

🛟
PM 关键洞察

失败恢复不是售后逻辑，而是主路径能力。Agent 产品越强，越必须假设它会在中途遇到长上下文、预算、权限、工具异常等现实限制。

04

作为 AI Agent PM，你该带走什么

最后一层不是背架构图，而是把架构变成 PM 的思考框架：能力如何打包、风险如何控制、指标如何定义、失败如何产品化。

做个小测验

如果你要给 Claude Code 加一个“自动批量改配置”的新功能，最先应该从哪一层设计起？

四个 PM 级别的启示

1
能力设计

决定新能力是 command、tool、skill，还是 plugin，而不是只写一个入口说明。

2
信任设计

权限提示、自动批准规则、危险动作的确认点，都是产品的一部分。

3
恢复设计

compact、重试、摘要和 stop hooks 证明失败不是边缘场景，而是主路径的一部分。

4
指标设计

除了满意度，还要看任务完成率、平均工具回合数、恢复成功率、权限拦截率和单任务成本。

把这些启示落成 PM 的日常问题

?
能力放哪一层

这个需求是新 command、新 tool，还是新 skill？如果说不清，后面一定会乱。

?
什么动作可以默认发生

不只是“能不能做”，而是“在什么模式下允许自动做”。

?
失败后用户看到什么

是死掉，还是给出下一步建议，还是自动 compact 后继续？

?
团队靠什么判断系统在变好

如果没有 completion、recovery、cost 和 latency 的共同指标，路线图会只剩主观感觉。

05

系统如何知道自己能做什么

这一层经常被低估，但它决定了 agent 的质量上限和风险边界。上下文注入和权限上下文，实际上是产品策略落到代码里的地方。

context.ts 在做的事，不只是“补充信息”

它在决定模型是否拿到 git 状态、项目记忆、当前日期，以及何时禁止注入某些内容。换句话说，它在决定 agent 的世界模型到底有多完整。

Code 
const shouldDisableClaudeMd =
  isEnvTruthy(process.env.CLAUDE_CODE_DISABLE_CLAUDE_MDS) ||
  (isBareMode() && getAdditionalDirectoriesForClaudeMd().length === 0)

const claudeMd = shouldDisableClaudeMd
  ? null
  : getClaudeMds(filterInjectedMemoryFiles(await getMemoryFiles()))
Plain English

系统不是每次都盲目加载项目记忆，而是先判断当前模式是否允许这么做。

如果用户进入 bare mode，系统会更克制，避免自动带入没被明确要求的上下文。

这说明上下文不是“越多越好”，而是要和用户控制权一起设计。

权限上下文本质上是产品政策

Tool.ts 里定义的不是单纯技术类型，而是整个系统的允许边界。它决定 agent 什么时候能直接做、什么时候要问、什么时候必须拒绝。

Code 
export type ToolPermissionContext = DeepImmutable<{
  mode: PermissionMode
  additionalWorkingDirectories: Map<string, AdditionalWorkingDirectory>
  alwaysAllowRules: ToolPermissionRulesBySource
  alwaysDenyRules: ToolPermissionRulesBySource
  alwaysAskRules: ToolPermissionRulesBySource
  isBypassPermissionsModeAvailable: boolean
  isAutoModeAvailable?: boolean
}>
Plain English

每个会话都带着一份权限配置，不是临时拍脑袋决定。

这里同时定义了默认模式、永远允许、永远拒绝、永远先问等规则来源。

这相当于把 PM 和安全团队的策略直接编进了运行时契约。

alwaysAllowRules低摩擦、高频动作 alwaysAskRules需要用户确认 alwaysDenyRules强约束边界 mode会话级行为模式

对 PM 来说，这里对应三种设计权衡

更多上下文

通常会提升回答质量，但也会增加成本、引入不必要信息，甚至产生越权感。

更宽权限

通常会提升自动化程度，但会放大误操作半径，降低用户信任和可预测性。

更强约束

通常会提升安全感，但也可能让 agent 变得笨重、频繁打断、无法流畅完成任务。

更细模式

通常能兼顾效率和治理，但会增加学习成本，需要更好的界面和文案来解释差异。

06

指标、经济性和扩展面

最后一层是运营视角。一个 agent 产品不只是能跑，还要知道值不值得跑、什么时候该扩、扩到哪一层最合理。

tools.ts 暴露的是能力地图，也是成本地图

每多一个工具，系统就多一块能力，也多一块风险、观测和成本面。好的 PM 会把“新增能力”当成“新增运营责任”。

Plain English

这里列的不是菜单，而是模型这一轮理论上可以使用的能力边界。

每个工具都可能带来新的成功路径，也带来新的失败模式和权限问题。

所以“加一个工具”从来不是简单加功能，而是扩张整套系统的行为半径。

一套更像 AI Agent PM 的指标面板

Completion

任务完成率

不是“模型回答了”，而是“用户想做的事真的完成了”。

Recovery

恢复成功率

上下文超长、工具失败、权限阻断后，有多少任务最终还是被拉回完成态。

Cost

单任务成本

结合 tokens、工具回合和外部调用，看自动化是否真的划算。

Latency

首反应时延

启动和首轮响应速度决定用户是否愿意频繁使用 agent。

Trust

权限接受率

如果大量权限提示被拒绝，说明边界设计和默认模式有问题。

Surface

扩展成功率

新 command、skill、plugin 上线后，是否真的被复用，还是只是在增加复杂度。
