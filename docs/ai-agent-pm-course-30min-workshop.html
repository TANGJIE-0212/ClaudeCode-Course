03 7 min runtime

一次请求是怎么被推进、被打断、又被救回来的

这部分是整个 workshop 的心脏。你不需要背 API 细节，但你必须理解一轮任务为什么不是“一问一答”，而是一个会持续推进的状态机。

query.ts 用的不是普通函数，而是异步生成器

code translation
Code 
export async function* query(
  params: QueryParams,
): AsyncGenerator<
  | StreamEvent
  | RequestStartEvent
  | Message
  | TombstoneMessage
  | ToolUseSummaryMessage,
  Terminal
> {
  const consumedCommandUuids: string[] = []
  const terminal = yield* queryLoop(params, consumedCommandUuids)
  for (const uuid of consumedCommandUuids) {
    notifyCommandLifecycle(uuid, 'completed')
  }
  return terminal
}
Plain English

这不是“发一个请求然后等结果”的函数，而是一个边执行边吐出事件的循环。

真正的工作在 queryLoop 里发生，它会根据模型结果决定是否继续调工具、继续追问、继续恢复。

任务结束时，系统还会把本轮相关命令统一标记成 completed，保证会话状态一致。

自学版逐行拆解

line by line
1
export async function* query(

这一行同时声明了三件事：它是对外暴露的函数、它里面会有异步行为、它不是一次性返回结果，而是会一边执行一边产出内容。

2
params: QueryParams,

这里把这轮任务要用到的输入都收进一个统一参数对象里。这样 query 不需要依赖一堆零散变量，而是明确地拿到 messages、context、toolUseContext、budget 等运行材料。

3
): AsyncGenerator<...>

这一段类型声明在告诉你：这个函数产出的不是单一字符串，而是一串事件和消息流，最后才返回一个 Terminal 结果。所以它天然适合流式 UI 和多步 agent 运行时。

4
const consumedCommandUuids: string[] = []

先准备一个数组，用来记录这轮任务里已经消费掉的 command。你可以把它理解成“本轮执行流水账”，后面结束时要统一补状态。

5
const terminal = yield* queryLoop(...)

真正的主工作委托给 queryLoop。这里的 yield* 很关键，意思是把 queryLoop 产出的事件原样继续往外转发，所以外层调用者会觉得自己一直在直接消费主循环。

6
for (const uuid of consumedCommandUuids)

主循环结束后，再遍历这一轮记下来的 command id。也就是说，收尾动作不是散落在中间，而是集中在 query 结束时统一做。

7
notifyCommandLifecycle(uuid, 'completed')

这行是在补“命令生命周期已完成”的状态。产品上看，这是为了保证外部观察者、UI 和状态系统都知道：这一轮里相关命令真的结束了。

8
return terminal

前面流式吐出的是过程，这里返回的是最终终态。也就是说，这个函数同时兼顾了“过程可见”与“结果可收束”两件事。

恢复逻辑决定产品是不是“真能用”

recovery

下面这段代码非常 PM。它说明作者预期系统会遇到 max_output_tokens 这种中途失败，而且不是“失败就算了”，而是要小心决定是否继续恢复，避免让上层消费者过早认为任务结束了。

Code 
const MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3

function isWithheldMaxOutputTokens(
  msg: Message | StreamEvent | undefined,
): msg is AssistantMessage {
  return msg?.type === 'assistant' && msg.apiError === 'max_output_tokens'
}
Plain English

系统知道自己可能因为输出太长中断，而且把这种情况当成常规运行条件，而不是罕见事故。

它还专门定义了恢复上限，避免无限重试，也避免把中间错误太早抛给上层产品。

这正是 agent 体验的关键分界线：是“一失败就死”，还是“遇到阻力也尽量把事做完”。

🛟
PM 关键洞察

恢复能力不是售后功能，而是主路径功能。用户不会记住系统第一次说得多好，但会记住它在中途出错时还能不能继续把任务做完。

04 5 min governance

上下文、权限和边界，就是产品政策

如果你把上下文注入和权限控制当成底层细节，那你会错过 AI Agent PM 最核心的一组设计杠杆。

context.ts 决定模型知道什么，不知道什么

context strategy
Code 
const shouldDisableClaudeMd =
  isEnvTruthy(process.env.CLAUDE_CODE_DISABLE_CLAUDE_MDS) ||
  (isBareMode() && getAdditionalDirectoriesForClaudeMd().length === 0)

Plain English

系统不是每次都盲目带入项目记忆，而是先判断当前模式和用户要求。

进入 bare mode 后，它会刻意克制，避免自动注入未经要求的信息。

这说明上下文不是越多越好，而是必须和用户控制权、成本、信任感一起设计。

更多上下文

通常提高质量，但也可能提高成本、带来无关信息，甚至让用户感觉越权。

更少上下文

通常更可控、更轻，但也会让 agent 频繁显得“没看懂场景”。

Workshop 1: 给一个新能力设计边界

3 min exercise
练习一

如果你要新增“自动批量改配置”，你会先设计什么？

请在脑子里或和同事快速讨论这四件事：1) 它是 command、tool 还是 skill；2) 哪些动作默认允许；3) 哪些动作必须确认；4) 失败后用户应该看到什么。

✅
参考答案

更合理的起点通常是 command + tool policy 的组合：用户通过显式入口触发，底层可能调用文件编辑或命令执行工具；默认只允许低风险配置改动，涉及跨目录、批量覆盖、服务重启时必须确认；失败时要告诉用户哪些配置已改、哪些没改、怎么回滚。

System Prompt 不是一段文案，而是一套装配系统

system prompt
Code 
if (overrideSystemPrompt) {
  return asSystemPrompt([overrideSystemPrompt])
}

if (feature('COORDINATOR_MODE') && ... && !mainThreadAgentDefinition) {
  return asSystemPrompt([
    getCoordinatorSystemPrompt(),
    ...(appendSystemPrompt ? [appendSystemPrompt] : []),
  ])
}

return asSystemPrompt([
  ...(agentSystemPrompt
    ? [agentSystemPrompt]
    : customSystemPrompt
      ? [customSystemPrompt]
      : defaultSystemPrompt),
  ...(appendSystemPrompt ? [appendSystemPrompt] : []),
])
Plain English

这套系统不是拿一段固定 prompt 到处复用，而是在按角色和模式做优先级装配。

协调者模式、主线程 agent、自定义 prompt、默认 prompt、附加 prompt 都有明确层级。

这意味着 PM 可以把 prompt 当成一种产品装配面，而不是只当一份写作素材。

它的特点

第一，支持 replace 和 append 两种装配关系。第二，coordinator 和普通 agent 的 prompt 身份不同。第三，system context 和 user context 是外接层，不直接被硬编码成一段不可拆的文本。

产品含义

这让 prompt 从“写得好不好”升级成“模式切换、角色切换、能力切换时如何稳定装配”。如果以后做 enterprise mode、review mode、proactive mode，这种结构非常值钱。

Agent 协作模式，本质上是 coordinator-worker 操作系统

multi-agent
Code 
You are a **coordinator**. Your job is to:
- Help the user achieve their goal
- Direct workers to research, implement and verify code changes
- Synthesize results and communicate with the user

Parallelism is your superpower.

Verification means **proving the code works**, not confirming it exists.
Plain English

这里已经不是“一个主 agent 偶尔拉几个助手”，而是明确把协调、实现、验证拆成不同角色。

更重要的是，系统要求 coordinator 先综合，再分派；要求 verifier 保持 fresh eyes，而不是顺手盖章。

这是一种组织结构，不只是并发技巧。

为什么这很强

它把 research、implementation、verification 设计成明确工序，还规定什么时候应该继续同一个 worker，什么时候应该起一个全新 worker，避免上下文污染。

对 PM 的启示

多 agent 协作不该只问“能不能并发”，而要问“谁负责综合、谁负责执行、谁负责反证、结果怎么回流”。这套代码已经把这些问题产品化了。

上下文压缩模式，不是缩短历史，而是分层保真

context compression
Code 
const reactiveCompact = feature('REACTIVE_COMPACT')
  ? require('./services/compact/reactiveCompact.js')
  : null

using pendingMemoryPrefetch = startRelevantMemoryPrefetch(
  state.messages,
  state.toolUseContext,
)

const isMainThreadCompact =
  querySource === undefined ||
  querySource.startsWith('repl_main_thread') ||
  querySource === 'sdk'
Plain English

压缩不是一个“触发了就总结一下”的单点动作，而是查询循环、相关记忆预取、主线程隔离清理共同组成的一条流水线。

系统既要压缩，又要保证 subagent compact 不会误伤主线程状态。

这说明他们在管理的是长期任务可持续性，而不是短会话漂亮演示。

它的模式

前面有 memory prefetch，中间有 reactive compact，后面还有 post-compaction cleanup。也就是说它是“预处理 + 触发压缩 + 善后恢复”的三段式设计。

为什么高级工程师会觉得好

因为它没有把 compact 写成一个粗暴的全局 reset，而是认真区分 main-thread、subagent、cache、memory files、system prompt sections 这些不同后果域。

记忆整理不是附属功能，而是 agent 定义的一部分

memory organization
Code

一层是在主会话里自动读取和注入项目记忆，另一层是在 agent 创建时决定要不要把“如何积累记忆”写进 agent 的系统说明。

这说明记忆既是 retrieval 问题，也是 agent behavior design 问题。

所以外界说它像 autodream 或 memory organizer，并不完全是空穴来风。

它整理了什么

项目级记忆、会话级记忆、agent scope memory、相关记忆预取、compaction 后的记忆再注入，形成了一条比较完整的 memory pipeline。

产品含义

这意味着未来 agent 的竞争力，不只是会不会调工具，而是谁能更稳定地形成 institutional knowledge，并在下一轮任务里拿回来继续用。

Surface

Claude Code 对 Harness 的 6 个启示

pm lens

如果最近你在看各种 agent harness，这个仓库最值得学的不是“怎么挂几个 tool”，而是怎样把模型包进一个可治理、可恢复、可观测的运行时。

1. Harness 首先是 runtime

query.ts 用循环推进任务、处理中断和恢复，所以系统不是一问一答，而是一个可持续运行的状态机。

2. 能力池要和可见菜单分开

tools.ts 先维护基础能力池，再按权限、环境、模式过滤出当前可见工具。这比“每个任务临时拼一套工具”稳定得多。

3. 协议层必须独立

Tool.ts 把权限上下文、执行上下文、共享契约单独抽出来。这样工具平台才能扩展，不会被注册逻辑绑死。

4. 边界治理比能力更重要

自动模式、permission context、working directory、sandbox refresh 这些机制说明，生产级 harness 卖的是“可控执行”，不是“无限执行”。

5. 下一站是多 worker

coordinator / worker / scratchpad 这些设计表明，成熟 harness 会原生支持分工、委派和隔离，而不是只做单 agent 壳。

6. 没有观测就没有产品

token、cost、编辑决策、session 指标都被纳入计量。没有这些，harness 只能 demo；有这些，PM 才能运营。

🧰
工具集合是不是根据每个任务都不一样？

对，但更准确地说：当前可见工具会因为任务形态、权限模式、环境状态、feature flag、是否连接 MCP 或 worker 模式而变化；但它们通常来自一个相对稳定的基础能力池。也就是说，harness 不是“每来一个任务就重造一套工具系统”，而是“在同一套能力底座上做动态裁剪”。

🧭
PM 该怎么用这 6 个启示

以后看一个 harness，不要只问“支不支持 tool calling”。先问 4 件事：它有没有持续运行 loop、有没有权限和 sandbox 边界、有没有恢复路径、有没有成本与效果观测。能回答这 4 个问题的，才更接近产品；答不出来的，大概率还只是 demo。
