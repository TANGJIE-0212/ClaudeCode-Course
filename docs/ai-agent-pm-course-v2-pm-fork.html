Claude Code 的产品骨架

从 PM 视角看，它可以拆成四层：入口、编排、执行、治理。这个分层不是学术上的整洁，而是影响功能如何被扩展、如何被授权、如何被运营。

入口层

commands.ts 决定用户如何显式触发任务。它是产品导航和任务模板层。

编排层

query.ts 决定这轮请求如何被推进、恢复、收束，是 agent runtime 的大脑。

执行层

tools.ts 和 Tool.ts 把模型可调用的现实动作变成可治理的原子能力。

治理层

权限模式、上下文注入、恢复路径、预算控制，决定系统能否被信任。

PM 应该问的不是“能不能做”，而是“应该落在哪层”

New command?
当用户需要一个清晰、可发现、可复用的顶层入口时，用 command。
New tool?
当模型需要一个底层动作，且这个动作要可授权、可审计、可组合时，用 tool。
New skill?
当你要沉淀一整套可复现的工作流，而不是一个原子动作时，用 skill。
New plugin?
当能力需要跨团队扩展或接入外部生态时，用 plugin 或 bridge。

把一次需求拆成 PM 能操作的运行时步骤

如果用户说“帮我 review 这个仓库”，你应该能把它拆成下面这些运行时阶段，而不是只想到“发给模型”。

1
确定入口

这是自由输入、slash command，还是某个 skill 的典型场景？入口不同，预设上下文和交互预期就不同。

2
准备上下文

需要 git 状态吗？需要项目记忆吗？需要会话恢复吗？需要特殊权限模式吗？

3
规划执行

模型要不要先搜索？要不要开子代理？要不要进入 plan mode？这些决定了任务效率和可控性。

4
约束风险

哪些工具能自动调用，哪些要确认，哪些在特定环境必须阻断？

5
设计恢复

如果上下文爆了、输出超限、工具失败，系统怎么继续？好的 agent 体验总有 recovery story。

为什么这题重要

普通 PM 题会问“这个功能好不好用”。AI Agent PM 题要问“它在什么边界内被允许发生”“失败后怎么恢复”“谁来承担决策和责任”。

这也是这份 PM fork 和上游原版 skill 的差别：它不只是帮你看懂代码，而是逼你把代码变成产品操作系统。

这三个版本该怎么比较

你现在手上会有三个不同取向的产物：一个偏演讲 deck、一个偏 upstream 风格的互动课程、一个偏 PM notebook 的课程化版本。
