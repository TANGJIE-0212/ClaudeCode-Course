核心代码：AsyncGenerator 查询循环

code walkthrough
query.ts 
export async function* query(
  params: QueryParams,
): AsyncGenerator<
  | StreamEvent
  | RequestStartEvent
  | Message
  | ToolUseSummaryMessage,
  Terminal
> {
  const consumedCommandUuids: string[] = []
  const terminal = yield* queryLoop(
    params, consumedCommandUuids
  )
  for (const uuid of consumedCommandUuids) {
    notifyCommandLifecycle(uuid, 'completed')
  }
  return terminal
}
人话翻译

async function* — 不是一次性返回结果，而是边执行边吐出事件流。

yield* queryLoop — 把主循环的所有事件原样转发给上层消费者。循环内部会根据模型输出决定是否继续调工具。

notifyCommandLifecycle — 任务结束时统一补状态，保证外部观察者知道任务真的完成了。

这个函数同时实现了「过程可见」和「结果可收束」两件事。

恢复逻辑：产品是不是「真能用」的分界线

recovery system
恢复代码 
const MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3

function isWithheldMaxOutputTokens(
  msg: Message | StreamEvent | undefined,
): msg is AssistantMessage {
  return msg?.type === 'assistant'
    && msg.apiError === 'max_output_tokens'
}
产品含义

系统把「输出太长被截断」当成常规运行条件，不是罕见事故。

最多重试 3 次，避免无限循环，也避免把中间错误过早暴露给用户。

其他恢复路径：自动压缩（上下文 >90% 时总结历史）、工具结果截断（>100KB 落盘）、瞬态错误重试。

🛟
PM 洞察

用户不会记住系统第一次说得多好，但会记住它在中途出错时还能不能继续把任务做完。恢复能力是主路径功能，不是售后功能。

关键系统设计：六个值得细看的控制面

1. 权限治理系统

permissions
ToolPermissionContext 
export type ToolPermissionContext = DeepImmutable<{
  mode: PermissionMode
  alwaysAllowRules: ToolPermissionRulesBySource
  alwaysDenyRules: ToolPermissionRulesBySource
  alwaysAskRules: ToolPermissionRulesBySource
  isBypassPermissionsModeAvailable: boolean
  isAutoModeAvailable?: boolean
}>
产品含义

每个会话带着一份「权限契约」运行，不是自由发挥。

三条规则链：始终允许（高频低风险）、始终拒绝（平台红线）、始终询问（需确认）。

权限匹配器用闭包预编译正则，一次解析、多次匹配。

default 默认模式 auto 自动模式 ask 逐一确认 bypass 跳过（受限）

2. System Prompt 装配系统

prompt assembly
prompt 优先级 
if (overrideSystemPrompt)
  return asSystemPrompt([overrideSystemPrompt])

if (feature('COORDINATOR_MODE') && ...)
  return asSystemPrompt([
    getCoordinatorSystemPrompt(),
    ...(appendSystemPrompt ? [appendSystemPrompt] : []),
  ])

return asSystemPrompt([
  ...(agentSystemPrompt
    ? [agentSystemPrompt]
    : customSystemPrompt
      ? [customSystemPrompt]
      : defaultSystemPrompt),
  ...(appendSystemPrompt ? [appendSystemPrompt] : []),
])
设计思路

Prompt 不是一段固定文案，而是按角色和模式做优先级装配。

支持 replace（整体替换）和 append（追加）两种装配方式。

Coordinator、Agent、Custom、Default 四个身份有明确层级。

这让 prompt 从「写得好不好」升级成「不同模式下如何稳定装配」。

06harness insights

对 Harness Engineer 的 6 个启示

如果你在构建或评估 Agent Harness，这 6 个点比「支不支持 tool calling」重要 10 倍。

1. Harness 首先是 Runtime

query.ts 用循环推进任务、处理中断和恢复。系统不是一问一答，而是可持续运行的状态机。

2. 能力池 ≠ 可见菜单

tools.ts 先维护基础能力池，再按权限、环境、模式过滤出当前可见工具。「动态裁剪」而不是「每次重造」。

3. 协议层必须独立

Tool.ts 把权限上下文、执行上下文、共享契约单独抽出。这样平台能扩展，不被注册逻辑绑死。

4. 边界治理 > 能力多少

auto-mode、sandbox、permission context、working directory 这些机制说明，卖的是「可控执行」不是「无限执行」。

5. 下一站是多 Worker

coordinator / worker / scratchpad 设计表明，成熟 harness 原生支持分工、委派和隔离。

6. 没有观测就没有产品

token、cost、编辑量、session 指标都被纳入计量。没有这些，harness 只能 demo；有这些，PM 才能运营。

🧭
评估 Harness 的 4 个核心问题

① 有没有持续运行 loop？ ② 有没有权限和 sandbox 边界？ ③ 有没有恢复路径？ ④ 有没有成本与效果观测？能回答这 4 个问题的，才更接近产品；答不出来的，大概率还只是 demo。

不是简单的 hjkl 移动，而是一个完整的 Vim 状态机：Normal / Insert / Replace / Visual 模式，支持 f/F/t/T 查找、text objects、dot-repeat（记录 insertedText 用于 . 命令）、g 前缀命令。

🌐

Bridge Mode（远程执行）

31 个文件构成的远程执行系统。JWT 认证、设备信任验证、加密会话凭据。支持 Stdio / SSE / HTTP / WebSocket 四种传输。本质上是把 Claude Code 变成了一个远程开发环境。

🌳

Git Worktree 隔离

Agent 可以在独立的 git worktree 里工作，不影响主分支。实验性代码修改在隔离环境执行。这是「可控实验」的基础设施。

⏰

Cron 定时任务

ScheduleCron 工具让 Agent 可以设置定时触发。不再需要人手动启动——Agent 可以自动定期执行巡检、构建、监控等任务。

🔙

Thinkback Play（推理回放）

/thinkback-play 命令可以回放 Agent 的思考过程。对 debug 和理解 Agent 决策路径非常有价值。

最后：一张 PM 带走的 Checklist

takeaway

以后每次你提一个 Agent 新功能，都先问这 5 个问题：

1

落在哪一层？

Command / Tool / Skill / Plugin

2

需要哪些上下文？

Git / 记忆 / 权限 / 会话

3

默认能到哪一步？

Allow / Ask / Deny

4

失败怎么办？

恢复 / 提示 / 接管

5

怎么知道变好了？

Completion / Cost / Trust

🧭
带走的终极判断题

如果一个新功能提案只讲「模型会帮用户做什么」，却讲不清楚它落在哪一层、怎么被授权、出错怎么恢复、如何被观测，那它还不是一个成熟的 AI Agent PM 提案。
